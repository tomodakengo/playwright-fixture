# Add Authenticated State

Pick the right pattern from the table, then follow the recipe.

## Pattern matrix

| User intent | Pattern | When |
|---|---|---|
| "Tests should start logged in as `<role>`" | **Pattern 1** — `storageState` from setup project | Default for "logged-in" tests |
| "Login is slow / I'm testing post-login flows, not login itself" | **Pattern 2** — API login (`loginViaApi`) | Tests that just need a session, fast |
| "Switch from `standard` to `admin` mid-test" | **Pattern 3** — `app.switchUser('admin')` | Multi-role flows |
| "Test the login form itself" | **Pattern 4** — drive the form via `app.login()` | Login UX tests |

## Pattern 1 — `storageState` (most common)

1. If the role does not exist yet, add it to `test-data/users.ts` (and `src/app/data/users.ts` if extending the demo app).
2. Add the role to the loop in `examples/tests/auth.setup.ts`:

   ```ts
   const rolesToAuthenticate: RoleName[] = ['admin', 'standard', 'newRole'];
   ```

3. In the spec:

   ```ts
   import { test, expect } from '@fixtures/fixtures';
   import { storageStatePath } from '@fixtures/auth';

   test.use({ storageState: storageStatePath('newRole') });

   test('should ...', async ({ app }) => { /* already logged in */ });
   ```

4. Confirm `tests/.auth/` is gitignored (it already is). Verify with `npm test`.

## Pattern 2 — API login (fastest)

```ts
import { test, expect } from '@fixtures/fixtures';
import { loginViaApi } from '@fixtures/auth';

test('should ... (logged in via API)', async ({ app, context, request }) => {
  await loginViaApi(request, context, 'standard');
  await app.products().goto();
  /* ... */
});
```

Use this when the test is not about login UX — it skips the form rendering entirely.

## Pattern 3 — Switch role mid-test

```ts
test.use({ storageState: storageStatePath('standard') });

test('should access admin page after switching to admin', async ({ app }) => {
  await app.products().goto();
  await app.switchUser('admin');   // creates a new context+page authenticated as admin
  await app.admin().goto();
  await expect(app.admin().title).toBeVisible();
});
```

`switchUser` works because the `App` facade reads its `Page` through a getter that points at the `store.ts` singleton. After `switchUser`, the next `app.<area>()` call returns a PageObject bound to the new page.

## Pattern 4 — Drive the login form (login UX tests)

```ts
test('should show error on bad credentials', async ({ app }) => {
  await app.login().goto();
  await app.login().signIn('standard', 'wrong-password');
  await expect(app.login().errorAlert).toContainText(/invalid/i);
});
```

## After ANY change

- Run `npm run typecheck && npm run lint && npx playwright test`.
- If you added a new role: confirm `tests/.auth/<role>.json` is generated by the setup project (it shows up after the first run).
